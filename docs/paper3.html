Governance, Earned Emission,
and the Remote-Execution Boundary

AnEntrypoint
April 2026
Abstract
We describe four further advances to the gm orchestrator, extending the lifecycle control of [1] and the in-phase discipline of [2]. First, we introduce a governance layer that separates route discovery, weak-prior handoff, and emission legitimacy into three distinct responsibilities with orthogonal state planes; the layer encodes a sixteen-mode failure taxonomy, an eight-case stress suite, and quality metrics (ΔS, λ, ε, coverage) that gate the UNKNOWN→KNOWN transition. Second, we reclassify git push as a remote-execution event: every push triggers continuous-integration workflows whose results are ground truth, and a push not followed by a CI watch is an unwitnessed execution in violation of the mutable contract. Third, we document a class of chronic zero-job CI failures rooted in silent YAML validation, and the remediation patterns for four common shapes. Fourth, we strengthen fact persistence: the memorize handoff fires at the moment an unknown resolves, not at phase exit, with an end-of-turn self-check for missed resolutions. Together these advances close gaps between agent narrative and ground truth.

1. Introduction

Paper I [1] introduced the phase-gated state machine; paper II [2] addressed the quality of work within phases. Both assumed that an agent operating within the prescribed envelope would produce correct results. Experience shows this assumption fails in four distinguishable ways:

  1. Plausibility-as-authorization: An agent that has successfully enumerated routes through a problem space treats the most plausible route as authorization to emit a solution. Plausibility is weaker than witnessing; treating it as stronger produces confident but unearned output.
  2. Silent remote execution: Every git push triggers GitHub Actions workflows that run real code against real state. An agent that pushes and does not observe the resulting workflow is executing code remotely without witnessing the execution—a category violation of the mutable contract.
  3. Parse-time CI failures: Workflow files can be syntactically valid YAML while still failing GitHub’s stricter validation, producing runs that complete in zero seconds with zero jobs dispatched and no logs. Every push registers the same failure; no job ever runs; the agent may believe it has deployed something.
  4. Compaction-era forgetting: Facts resolved mid-session die on context compaction unless persisted. Deferring the memorize handoff to phase end leaves a window in which resolved unknowns evaporate and must be re-discovered.

This paper describes the discipline advances addressing each. §2 introduces the governance layer; §3 addresses the remote-execution boundary; §4 catalogues CI failure modes; §5 tightens fact persistence; §6–8 discuss related work, limitations, and conclusions.

2. The Governance Layer

2.1 Three Responsibilities, Not One

Earlier work [1, 2] treated each phase of the state machine as owning a single responsibility: PLAN enumerates, EXECUTE witnesses, EMIT writes, VERIFY confirms. In practice, three distinct failure surfaces coexist at every phase and must be respected simultaneously:

Route discovery:
Where could this fail? What family of fault does the work live in? Owned by PLAN but referenced throughout.
Weak-prior bridge:
Information flowing forward from earlier phases arrives as weak prior, not authorization. Owned by EXECUTE.
Legitimacy gate:
Did this answer earn its requested strength? Owned by EMIT and VERIFY.

Neither route-first nor legitimacy-first alone suffices. The weak-prior bridge exists precisely to prevent route plausibility from silently converting into authorization as the agent advances through phases.

2.2 Seven Route Families

Every planned item is classified into at least one of seven route families:

FamilyWhat breaks here
groundingRetrieval, lookup, fact anchor
reasoningInference chain, logic, derivation
stateMemory, persistence, session continuity
executionRuntime, scheduling, process lifecycle
observabilityInspection, tracing, debuggability
boundaryInterfaces, contracts, subsystem seams
representationData shape, schema, type

The family is recorded in the .prd item. Its purpose is to discipline the repair move: a fault in the grounding family does not accept a reasoning fix; a fault in boundary does not accept a state fix. Mis-routed repair wastes an EXECUTE pass and forces regression to PLAN.

2.3 Four State Planes

Any in-flight item occupies four orthogonal state planes simultaneously. Advancement on one plane does not advance any other:

  • route_fit: unexamined → examined → dominant. Examined is not dominant. Dominant is not authorized.
  • authorization: none → weak_prior → witnessed. Only witnessed permits emission.
  • repair_legality: unverified → local_candidate → structural. Local candidate cannot ship as structural repair.
  • hidden_decision_posture: open → down_weighted → closed. Closing before CI green is illegal.

Orthogonality is the point. An item may have route_fit=dominant and still have authorization=weak_prior. Collapsing these planes—treating a dominant route as an authorized conclusion—is the primary governance failure.

2.4 Five Refused Collapses

A conclusion ships only when none of the following has occurred:

  1. Route collapsed into authorization (“the plan looks good” became “the code is right”).
  2. Candidate repair collapsed into structural repair (local patch presented as architectural fix).
  3. Hidden orchestration collapsed into public law (internal convenience shipped as contract).
  4. Cleanliness collapsed into legitimacy (code-compiles treated as evidence-supports).
  5. One strong route collapsed into universal closure (best available answer treated as only possible answer).

These are not general rhetorical hazards—they are the specific failure shapes the governance layer is designed to refuse. Each corresponds to a plane of collapse between the four state planes.

2.5 Quality Metrics

Before a mutable’s status flips UNKNOWN→KNOWN, four quality metrics are applied:

  • ΔS (drift): semantic delta between expectation and witnessed output. ΔS ≠ 0 means the mutable remains open, regardless of narrative framing.
  • λ (convergence): have two independent paths (different search, different import, different caller) reached the same answer? λ = 1 is single-witness; the mutable is still unknown.
  • ε (harmony): do adjacent invariants still hold—neighboring callers, types, tests? ε violated is local fix with side effect.
  • Coverage ≥ 0.70: for retrieval mutables, fraction of relevant corpus inspected. Below threshold, grounding has not been earned.

These are verbal checks, not machine-evaluated numbers. Their purpose is to give the agent a vocabulary for why a mutable is not yet resolved, so the narrative cannot silently skip the check.

2.6 The Sixteen-Mode Failure Taxonomy

Every fault surface enumerated during planning is mapped against a sixteen-mode taxonomy: hallucination & chunk drift, interpretation collapse, long reasoning drift, bluffing, semantic-embedding mismatch, logic collapse requiring reset, cross-session memory breaks, debugging black box, entropy collapse, creative freeze, symbolic collapse, philosophical recursion, multi-agent chaos, bootstrap ordering, deployment deadlock, and pre-deploy collapse. A fault surface that maps to none of the sixteen is either a seventeenth mode (and the taxonomy must be extended) or the surface is not yet named sharply enough. Items with no taxonomy mapping ship silent bugs.

2.7 Governance Stress Suite

Before declaring a non-trivial task complete, the proposed change is mentally walked through an eight-case stress suite that exercises forced-closure pressures: missing evidence, unsourced figures, ambiguous clauses, contradictory witnesses, attribution under time pressure, root-cause analysis with live alternatives, authenticity assessment from partial signals, and deploy-gate decisions under CI flake. The legal outcomes are: illegal commitments = 0; evidence-boundary violations = 0; lawful downgrades available = 8; outlier visibility preserved. A case the change flunks blocks completion.

Principle (Earned emission). A plausible-looking answer that has not cleared route_fit, authorization, repair_legality, and the stress suite is not eligible for emission. Lawful downgrade—writing a weaker, true statement in place of a stronger, unearned one—is always available; forced closure never is.

3. The Remote-Execution Boundary

3.1 Git Push as Unwitnessed Execution

Every git push triggers GitHub Actions workflows. Those workflows build binaries, publish packages to npm, deploy GitHub Pages sites, trigger downstream repository updates, and commit artifacts back to the repository. Each is execution of real code against real infrastructure. The agent that pushes and does not observe the resulting workflow is executing code without witnessing the execution—which is the precise failure the mutable contract forbids.

The reclassification is: git push is not a local operation. It is a remote-execution event. Every push requires a CI watch the same way every local hypothesis requires a witnessed probe. The watch is not an optional verification step deferred to the completion phase—it is the execution contract applied to remote actions.

3.2 Watch Protocol

Every push is followed immediately by:

  1. Enumerate the runs the push triggered: gh run list --limit 5 --json databaseId,name,status,conclusion.
  2. For each run in queued or in_progress, watch until terminal: gh run watch <id> --exit-status.
  3. On failure, inspect: gh run view <id> --log-failed. Classify the cause: file output wrong, logic wrong, or new unknown. Regress to the owning phase.
  4. Check downstream cascade: pushes to this repository may trigger workflows in downstream repositories. Each downstream cascade is watched the same way.

A push without a matching watch cycle is a contract violation. This rule supersedes the earlier expectation that CI could be reviewed at the completion phase—by completion, the downstream cascade has already happened.

3.3 The Cascade

A push to one repository is a local operation; a push that triggers cascading deployments is a multi-repository remote execution. Our particular case: a push to the orchestrator triggers plugin-repository builds across twelve downstream repositories, each of which may in turn publish a package to npm, deploy a site, and commit artifacts back to its own default branch. A failure at any stage of this cascade is a failure introduced by the original push.

The cascade motivates the strict version of the watch protocol. The window in which the agent can still fix a bad push narrows as artifacts propagate. An agent that notices a failure only at the completion phase has already shipped artifacts to twelve repositories and, in some cases, public package registries.

4. Parse-Time CI Failures

4.1 The Zero-Job Failure Signature

A specific class of GitHub Actions failure manifests as: workflow run created, status completed, conclusion failure, duration zero seconds, zero jobs dispatched, no logs available. The gh api workflow registry shows the workflow’s name field as the file path rather than the declared name, indicating that GitHub’s validator never successfully parsed the file’s top-level name: directive despite the file being syntactically valid YAML to local parsers.

This failure shape is silent in practice: every push produces the same zero-second failure, the workflow never actually runs, and an agent checking only that “all workflows completed” can believe its changes have been deployed when no job has executed in weeks.

4.2 Four Root Causes Catalogued

During a single debugging session we resolved four distinct shapes of this failure. Each is reproducible and remediable:

4.2.1 Block-scalar Indentation of Heredoc Content

A run: | block scalar with nested heredoc content has a subtle indentation requirement. The heredoc marker sits at the block-scalar base indent, but heredoc body lines in the source file must maintain at least that indent. If the body is at column zero (natural Python source), the YAML parser accepts the file as valid at top level but fails inside the block-scalar content, producing the zero-job failure. Remediation: indent the heredoc body to the block-scalar base indent. The wrapping run step strips that indent before invoking the shell, so the invoked python3 sees correctly column-aligned code.

4.2.2 Missing Token Environment Variable

Workflow steps that invoke curl against the GitHub API without passing GITHUB_TOKEN receive anonymous rate-limited responses. A rate-limited response is a JSON error object, not the expected list; downstream Python that iterates over the response crashes with an AttributeError. The fix is a step-level env: GITHUB_TOKEN: ${{ github.token }} plus an isinstance guard on the parsed response. Both are required; omitting either produces an intermittent failure that correlates with API rate-limit windows.

4.2.3 Concurrent-Push Races

Multiple scheduled workflows that commit to the same branch on every run will race when triggered by the same push. The first workflow wins; the rest receive ! [rejected] main -> main (fetch first). The generalized remediation is a retry loop around git push with jittered backoff and git pull --rebase between attempts:

for i in 1 2 3 4 5; do
  git pull --rebase origin main && git push && break \
    || sleep $((RANDOM % 10 + 5))
done

This is not specific to this project. It is the pattern for any CI-committed artifact on a shared branch.

4.2.4 Gitignored Output Paths

Workflows that generate artifacts into a gitignored directory fail silently: git add path/ returns exit 1 against an ignored path, and with pipefail enabled the step aborts. The remediation is git add -f for the specific path. Keeping the output in .gitignore prevents local developers from accidentally committing the artifact; force-adding in CI lets the workflow commit it when it is the authoritative producer.

4.3 A Fifth: Multiline Output Expansion

A separate failure shape arises when one step’s multiline output is expanded via ${{ steps.X.outputs.Y }} into a subsequent step’s shell script. If the output contains unescaped backticks, parentheses, or command substitutions, the expansion produces a bash syntax error at runtime. Remediation: write multiline content to a file, read it in the consuming step via jq --rawfile, and let jq handle JSON escaping. This pattern generalizes to any CI step that produces structured content for a later step.

Observation (Parse-time opacity). YAML validity in local parsers is not sufficient evidence of CI validity. The workflow registry’s handling of the name field is a witnessed probe of whether GitHub has accepted the file: when the registry shows the file path in place of the declared name, the file has failed GitHub’s validator and every run will fail at zero seconds until the file is changed.

5. Same-Turn Fact Persistence

5.1 The Compaction Window

An agent session accumulates context across many turns. When the context approaches its limit, earlier turns are summarized and replaced with the summary. Facts resolved in a turn but not persisted in durable storage before compaction are lost, and must be re-discovered in a later session. The background memorize sub-agent introduced in [2] handles persistence, but its invocation timing was previously loose: facts could be handed off at phase end rather than at resolution time.

In practice, the window between resolution and phase end is wide enough to lose facts to unrelated work: the agent resolves an unknown, moves to other tasks, accumulates context, and hits compaction before the phase exit at which the handoff was scheduled.

5.2 Same-Turn Handoff

The strengthened discipline is: the moment an unknown resolves, the memorize handoff fires in the same turn, before the next tool call where possible, end-of-turn at latest. The trigger contract enumerates the resolution events:

  • An exec: run’s output answers a prior “let me check”.
  • A code read confirms or refutes an assumption.
  • A CI log reveals a root cause.
  • The user states a preference, constraint, deadline, or decision.
  • A fix works for a non-obvious reason.
  • A tool or environment quirk bites once.

Each event fires a separate memorize call, in parallel when multiple events resolve in the same turn, never serialized. Every exit from a turn includes a self-check: any resolved unknown un-handed-off is handed off before the response closes.

5.3 Specification Documents as Code

A related advance concerns the search index. The exec:codesearch tool was previously limited to source files. Specification PDFs—protocol specs, datasheets, RFCs, compliance texts—are now indexed page by page alongside source. A constraint that appears only in a PDF is therefore a witnessable fact, not an unsourced assumption. A recommendation or emission that references a PDF constraint without a doc.pdf:<page> citation is classified as unwitnessed and must regress to EXECUTE.

This closes a common failure mode: the agent reading source code and reimplementing a protocol field based on an in-file comment or adjacent usage, when the authoritative definition was in a PDF the agent never searched.

6. Relationship to Earlier Work

The governance layer extends the mutable discipline of [1] orthogonally. Mutables are fact-level: an expected value, a current value, a resolution method. The four state planes are item-level: an item may have all its mutables resolved while still lacking authorization for emission. The separation handles the case where every individual fact is witnessed but their aggregate has not earned the strength of the conclusion being drawn from them.

The remote-execution boundary in §3 extends the witnessed-execution principle of [1] to actions that cross the machine boundary. Paper I described witnessed execution for local code; this paper extends the same contract to any action whose consequences happen elsewhere and whose correctness is knowable only through the feedback channel. git push is one such action; a future extension covers npm publish, package registry operations, and cross-repository API calls.

The same-turn memorize discipline in §5 tightens the timing of a mechanism introduced in [2]. The earlier paper established the background sub-agent; this paper establishes that its invocation cannot be deferred to phase end without accepting a loss window during compaction.

The CI failure taxonomy in §4 is a concrete instance of the governance layer’s insistence on naming the failure mode before applying a fix. Each of the four shapes has a name, a signature (the zero-job failure with name shown as path), and a reproducible remediation. Absent the naming, each instance appears as an idiosyncratic bug; with the naming, they are members of a single class of parse-time opacity failures addressable by a common protocol.

7. Limitations

  1. Governance as cognitive discipline: The four state planes and five refused collapses are enforced by agent self-discipline rather than mechanical gates. An agent that narrates past a plane collapse can still ship an emission. The stress-suite walk-through is likewise mental; it has no automated execution.
  2. CI watch cost: Watching every push serializes the agent against CI runtime. For long-running workflows, the agent is blocked; for cascade pushes, the blocking multiplies across downstream repositories. We accept this cost; a faster CI substrate would reduce but not eliminate it.
  3. Parse-time failures are not all catalogued: The four shapes in §4 were uncovered during a single debugging session. The underlying phenomenon—YAML files valid to permissive parsers but invalid to GitHub’s stricter validator—is not fully bounded. Additional shapes almost certainly exist and will require similar witnessed diagnosis.
  4. Memorize self-check: The end-of-turn self-check depends on the agent noticing un-handed-off resolutions. An agent whose attention is elsewhere at turn exit can still miss a resolution. The discipline reduces but does not eliminate the loss window.
  5. Sixteen-mode taxonomy is not closed: The taxonomy is adapted from external prior work and extended as new modes are encountered. Any fault surface the agent encounters and cannot map is presumptive evidence of a seventeenth mode rather than sufficient reason to emit without classification.

8. Related Work

The governance layer’s separation of route discovery, weak-prior transfer, and legitimacy gating is adapted from constitutional runtime architectures for AI reasoning systems [6], which distinguish plausibility (what seems right) from authorization (what has earned public emission). Our contribution is the operationalization of this separation in a coding-agent state machine, with specific state planes, metrics, and stress cases rather than the general constitutional framing.

The sixteen-mode failure taxonomy is likewise adapted from prior work on LLM system failure categorization [6], with route-family tags added to discipline repair moves. The quality metrics ΔS, λ, ε and coverage threshold are drawn from the same source.

The remote-execution boundary echoes the principle of observable effects [7]: any action whose consequences extend beyond the invoking process is an effect whose correctness must be observed externally. CI-as-remote-execution is a specific instance for push-driven build systems, where the workflow runner is the external observation point.

The parse-time CI failure catalogue has analogues in the compiler-error hierarchy [8]: errors at tokenization, parsing, type-checking, and linking are structurally distinct and require different remediations. YAML validity, CI-validator validity, step execution, and artifact production are four stages at which a workflow can fail silently; our catalogue is an empirical taxonomy of failures at the parsing boundary specifically.

9. Conclusion

We have described four advances: a governance layer separating route discovery, weak-prior handoff, and emission legitimacy into three responsibilities with four orthogonal state planes; a reclassification of git push as a remote-execution event with mandatory CI watch; a catalogue of four plus one parse-time CI failure modes with remediation patterns; and a strengthened same-turn memorize discipline with end-of-turn self-check.

These advances address failures that earlier work left unaddressed: not failures of phase order (paper I) nor failures of in-phase discipline (paper II), but failures of the agent’s understanding of its own authority to emit conclusions, and of the boundary between the agent’s local actions and their remote consequences. A well-structured lifecycle with disciplined in-phase work is still necessary and still not sufficient. The sufficiency gap remaining at the end of paper II was earned-emission and boundary-awareness. This paper addresses that gap.

Our experience suggests that agent reliability improves monotonically with the refusal to collapse plausibility into authorization. The governance layer is structurally equivalent to the mutable discipline of paper I applied one level up: individual facts are witnessed into KNOWN by execution; individual conclusions are witnessed into emittable by the governance layer’s plane advancements. The same principle, operating at a higher level of abstraction.

rs-exec

the runner AnEntrypoint/rs-exec ↗

Job: turn an exec:<lang> request into a real subprocess, stream its output back, and kill it when its budget is up. It is the muscle behind every line of code your agent claims to have run. If exec:nodejs works, rs-exec did the work. If exec:nodejs ate your script, that's a different rust crate's fault and rs-exec is offended you'd suggest otherwise.

what it actually does

  • runs an HTTP/JSON daemon on 127.0.0.1 with an OS-assigned port (no fixed port; Windows leaks orphan listeners on crash and a fixed port turns into a permanent allocation problem)
  • spawns the user's code in a subprocess with stdio piped, threads stream stdout and stderr back via per-chunk appendOutput RPCs
  • enforces an explicit timeout via tokio::time::timeout wrapping wait_for_completion — on elapse, walks the process tree via sysinfo and kills every descendant (Windows cmd.exe spawns survivor shells; on Unix bash pipelines escape the parent's group)
  • preserves partial streamed output on timeout so the agent sees what did flow before the kill
  • idle-cleans dead browser sessions and orphan exec-process-mode children every 60 seconds

commandments rs-exec enforces

  • mandatory timeoutMs on every execute RPC. missing or zero = hard error. no defaults. no warnings. the cushion has a clock.
  • streamed output is preserved through timeout-kill. the output array drains either way.
  • explicit timedOut: true field on the response. clients don't string-match the error.

what kills it

Windows port leaks. Solved by binding to port 0 and writing the live port to a single file atomically. Bun -e silently swallowing fs ENOENT throws. Solved by wrapping every nodejs/typescript snippet in a try/catch IIFE that explicitly writes to process.stderr. Long-running fetches stalled on tcp-alive but byte-dead sockets. Solved by a stall watchdog with a 20-second no-bytes deadline — the socket-idle timeout was useless because the connection was technically alive.

rs-codeinsight

the AST x-ray AnEntrypoint/rs-codeinsight ↗

Job: at session start, read the entire codebase and emit a single compressed overview — file inventory, top patterns, hub modules, dead code, complexity hotspots, language breakdown. The first 5–10 turns of every session used to be the agent re-discovering what's in the repo. rs-codeinsight gives that to it for free, in one paragraph, in token-dense format.

what it actually does

  • walks the project tree, parses every supported language via tree-sitter, builds a structural index
  • computes per-file metrics: lines, functions, classes, average complexity, parameter counts, long functions, large files
  • derives architectural layers (L0 pure exports → L3 pure imports), identifies hub modules and circular dependencies
  • flags issues: complex functions, large files, duplicated groups, SQL-injection literals, dead exports, orphaned files
  • caches results in .codeinsight so repeat session-starts are nearly free

what kills it

Repos with millions of lines time out the parse. The fix is to scope the analysis with --path, which is what gm-starter does on first run. Caching covers the rest.

rs-search

the file finder AnEntrypoint/rs-search ↗

Job: answer exec:codesearch <words>. natural-language semantic-style ranking over file content, with bm25 scoring, snippet windows, and tunable context. it's why the agent never types find and never reaches for rg: the dedicated tool is so much faster that there's no excuse for the alternatives.

what it actually does

  • scans the project (with respect for .gitignore), tokenizes file content, builds a sparse term-frequency index
  • scores queries via bm25 with snippet ranking; returns top matches with surrounding line context
  • handles PDFs as code (extracts text via pdf-extract, caches per-page in .gm/code-search/pdf-cache/)
  • exposes an MCP surface so non-CLI clients can hit it directly

commandments it enforces (via the hook layer)

  • blocked find, grep, rg, Glob, Explore — the agent gets a hook-deny redirecting to exec:codesearch
  • two words minimum query length to start; iterate by changing or adding one word per pass
  • four attempts before declaring something absent

rs-learn

the memory store AnEntrypoint/rs-learn ↗

Job: hold every fact the agent ever resolved, across every session, on every machine. exec:recall <query> reads from it. exec:memorize <tag> <fact> writes to it. The store is committed to the repo as .gm/rs-learn.db, so memory shares across machines, sessions, CI, and contributors. Knowledge is collective.

what it actually does

  • libsql-backed (sqlite-compatible) store of episodes; each episode has a source tag, content, embedding, classification
  • community detection (label propagation) groups related episodes; LLM summarization synthesizes per-community context
  • graph search via Searcher with configurable result count + relevance threshold
  • fast-path ingest (add_episode_fast) avoids re-embedding when content is identical
  • direct rust API consumed by rs-plugkit (no bun x rs-learn shim — the scoop bun.exe shim opens visible console windows on Windows)

commandments it enforces

  • tracked .gm/rs-learn.db is committed, never gitignored. memory must travel.
  • classified facts route to AGENTS.md (top-level rules) or rs-learn (everything else). classifier rejects changelog-shaped facts from AGENTS.md.
  • parallel memorize spawns are non-blocking background Agent(memorize) calls. multiple facts → multiple parallel Agent calls in one message.

rs-plugkit

the binary AnEntrypoint/rs-plugkit ↗

Job: the actual plugkit binary you have installed. it composes the other four crates plus the hook dispatcher. every PreToolUse, PostToolUse, Stop, SessionStart, UserPromptSubmit, and PreCompact hook is dispatched here, in rust, in microseconds, with deterministic exit codes that the editor uses to allow/deny tool calls.

what it actually does

  • routes plugkit hook <event> to the right handler (pre_tool_use, prompt_submit, stop, session_start, …)
  • reads .gm/turn-state.json to enforce per-turn compliance: gm-first, recall-before-benchmark, memorize counter
  • writes .gm/needs-gm sentinel on every prompt + every isMeta-bypassing event so the next tool call is forced through Skill(gm)
  • shells out to rs-exec for code execution, rs-codeinsight for the session-start overview, rs-search for codesearch, rs-learn for recall/memorize
  • auto-starts and supervises the rs-exec daemon; restarts on stale binary if no tasks are active

commandments it enforces

  • first action on every user message must be Skill(gm). no exceptions. isMeta-triggered responses still count.
  • blocked Bash(node|npm|npx|bun). Use exec:<lang>.
  • redirect find, grep, rg, Glob, Explore → exec:codesearch.
  • stop-block while .gm/prd.yml has items, while git is uncommitted, while there are unpushed commits.
  • memorize counter blocks after 3 consecutive exec calls without a Agent(memorize) spawn (write a sentinel to bypass with reason).

what kills it

Stale binaries. Solved by storing a stamp of the current exe path; on every health check, if the exe changed and no tasks are running, the runner restarts. Crashed daemons leave a port file pointing at a dead PID. Solved by writing the port file after the listener bound, atomically via rename, so clients see either the live port or a missing file (which they treat as "no runner").

How to update any of them

Push to the relevant repo. That's it. No cargo update, no local cargo build, no version bumps. The cascade pipeline detects the push, rebuilds the six prebuilt binaries, bumps gm-starter/gm.json::plugkitVersion, and triggers gm's twelve-platform fan-out. The next time anyone's editor calls /plugin, they get the new binary via bootstrap.js.

The PUBLISHER_TOKEN secret in rs-exec, rs-codeinsight, and rs-search is what authorizes the cascade.yml in those repos to trigger rs-plugkit's workflow_dispatch. If the cascade ever stops working, that token has rotated. Set it with gh secret set PUBLISHER_TOKEN --repo AnEntrypoint/<repo>.

Glootie's notes on the architecture

"Five crates is two more than necessary and three fewer than tempting. The temptation is to fold rs-search and rs-codeinsight into one — they both walk the tree. The reason not to: codeinsight runs once per session, search runs hundreds of times. Different latency budget, different cache shape, different release cadence. Sit down. Don't merge them." — Glootie, on cushion #4

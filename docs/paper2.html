Runtime Infrastructure for Autonomous
LLM Coding Agents

Browser Automation, Vector Search, and Cascade Deployment
AnEntrypoint
April 2026
Abstract
We describe a set of runtime infrastructure advances that extend the lifecycle control framework introduced in our earlier work on the gm orchestrator. The three central contributions are: (1) a deterministic managed-browser runtime for LLM agents that provides per-session Chrome isolation, automatic process cleanup, and cross-platform reliability without manual Chrome installation; (2) a hybrid code search backend combining BM25 term matching, vector embedding similarity (nomic-embed-text-v1.5), and git commit indexing, yielding both keyword precision and semantic recall; and (3) a fully automated cascade deployment pipeline that propagates any change in a Rust library repository through binary builds, version bumps, and plugin package publication without manual intervention. We also describe session-scoped task isolation with compound keys, orphan process reaping, a mandatory observability enumeration discipline, and a minimal-code structural mandate. These advances were validated in continuous production use across 11 agent platforms during March–April 2026.

1. Introduction

Our earlier paper [1] described the gm orchestrator—a phase-gated state machine for LLM coding agents that enforces lifecycle control through external hooks, named unknowns, and witnessed execution. That paper focused on the agent’s behavioral envelope: the rules that govern what an agent can do and in what order.

This paper describes the runtime infrastructure that makes those rules operational. Three problems motivated the work. First, agents that can execute arbitrary code need a reliable, isolated browser environment without depending on the user’s system configuration. Second, agents that search a codebase need both lexical precision (for exact identifiers) and semantic recall (for conceptual queries). Third, a multi-repository binary delivery pipeline that requires manual version management is a latent reliability failure—any step missed leaves agents running stale code.

The paper is organized as follows: §2 describes the browser runtime; §3 covers hybrid code search; §4 describes the cascade pipeline; §5 addresses session isolation and process hygiene; §6 covers the observability mandate; §7 introduces the minimal-code structural mandate; and §8–9 discuss limitations and related work.

2. Deterministic Browser Runtime

2.1 Problem

LLM coding agents frequently need to interact with web interfaces: testing frontend changes, scraping data, or automating browser-based workflows. The naive approach—invoking the user’s default Chrome installation via CDP—is unreliable: multiple agents share a single Chrome process and profile, producing cross-session contamination where one agent’s navigation state leaks into another’s. On Windows, this manifests as lock-file races on the Chrome user data directory when two agents attempt to launch simultaneously.

2.2 Architecture

The exec:browser runtime in rs-exec provides each agent session with an isolated Chrome instance. The runtime proceeds as follows on each invocation:

  1. Resolve a cwd-local browser profile path (.plugkit-browser-profile/), scoped to the working directory rather than the user account.
  2. Check whether an existing Chrome instance owns that profile and is reachable on the fixed debugging port (32882). If yes, connect to it.
  3. If no existing instance is found, clear any stale lock files from the profile, then launch a new managed Chromium (ms-playwright distribution) with --no-sandbox --disable-gpu --remote-debugging-port=32882.
  4. Wait up to 60 seconds for the profile lock to release if another agent holds it, then launch.

Session teardown is symmetrical: on session end, the plugkit stop hook kills the Chrome process, closes all playwriter sessions, and releases the profile lock. The exec:close command provides mid-session cleanup of individual tasks and their associated browser resources.

2.3 Session Ownership Enforcement

The rs-exec RPC layer enforces session ownership at the data layer. The handlers for getTask, getAndClearOutput, and waitForOutput reject requests from a session that did not create the task, returning an error rather than silently returning another session’s output. This closes the contamination vector at the protocol level rather than relying on naming conventions.

2.4 Reliability Engineering

Several Windows-specific issues required explicit handling:

  • Quote splitting: Playwriter’s -e argument uses shlex for quote-aware splitting, since Windows CMD does not handle single-quoted arguments the way POSIX shells do.
  • Port stability: The debugging port is fixed at 32882. A random port caused task-ID loss across runner restarts because the plugkit client could not reconnect.
  • Chrome exit code 21: Chrome exits with code 21 on Windows without the --no-sandbox flag; the managed browser launch unconditionally includes it.
  • Profile lock races: A profile held by another agent triggers a 60-second wait-and-retry before giving up and spawning a fresh instance.

2.5 drainSessionOutput and listSessionTasks

Two RPC methods were added to improve observability: drainSessionOutput flushes all buffered stdout/stderr for a session in one call (used by plugkit on every interaction to ensure no output is lost); listSessionTasks returns the full task list for a session with status, background flag, and last-seen output, providing an always-current view of what the agent has running.

3. Hybrid Code Search

3.1 Problem

Codebase search for LLM agents faces a precision-recall tension. BM25 term matching is precise for known identifiers but fails on conceptual queries (“find where we handle authentication errors”) where the vocabulary of the query and the vocabulary of the code diverge. Pure embedding search has the opposite problem: high recall on semantically related content but low precision on exact symbol names, and a tendency to surface conceptually adjacent but irrelevant code.

3.2 Four-Section Search Output

rs-search now produces structured output in four sections, each addressing a different query type:

BM25 symbol hits:
exact term matches in symbol names and identifiers
BM25 content hits:
exact term matches in file content
Vector similarity hits:
semantically similar code and documentation passages
Git commit hits:
commit messages matching the query terms

The agent receives all four sections simultaneously, combining BM25’s identifier precision with vector search’s semantic recall and commit search’s intent surfacing.

3.3 Vector Embedding Backend

The embedding backend uses nomic-embed-text-v1.5.Q4_K_M, a GGUF-quantized model distributed in six parts to stay within file-size limits. The model is loaded at first search and cached for the session lifetime. Embeddings are computed over code chunks and stored in a flat index; similarity is computed by cosine distance at query time.

The choice of nomic-embed-text-v1.5 reflects its design for code and technical text retrieval, unlike general-purpose embedding models that may underweight programming vocabulary. The Q4_K_M quantization provides a 4× size reduction relative to the full-precision model with negligible recall degradation on code retrieval benchmarks.

3.4 Git Commit Indexing

The git commit index provides the agent with change intent: a query for “session cleanup” surfaces not only code containing those terms but the commit messages that described the intent when the change was made. This is particularly valuable for understanding the rationale behind non-obvious code patterns that BM25 and vector search would surface without context.

3.5 Mandatory Two-Word Protocol

The codesearch skill enforces a minimum-two-word query protocol: single-word queries are rejected, requiring the agent to provide at least a second disambiguating term. This reduces result noise and prevents the most common failure mode of overly broad queries that return hundreds of hits from which the relevant result must be manually extracted. When no results are found, the protocol instructs the agent to change one word and add a third before concluding a symbol is absent—requiring a minimum of four attempts before declaring a negative result.

4. Cascade Deployment Pipeline

4.1 Problem

The gm system depends on five Rust repositories: rs-exec (code execution runtime), rs-codeinsight (code analysis), rs-search (search backend), rs-plugkit (binary entry point), and plugforge (plugin package generator). Previously, updating a Rust library required manually bumping version references, triggering builds, and verifying binary delivery through each downstream consumer. Any missed step left agents running stale code silently.

4.2 Pipeline Design

The cascade pipeline eliminates all manual steps:

  1. Trigger: A push to any of rs-exec, rs-codeinsight, or rs-search fires a workflow_dispatch to rs-plugkit via a PUBLISHER_TOKEN secret.
  2. Build: rs-plugkit’s release workflow runs cargo update (pulling the latest git dependency hashes for all upstream repos), then builds six platform binaries: linux/mac/win × x64/arm64.
  3. Version: rs-plugkit auto-bumps its own patch version in Cargo.toml and commits with [skip ci].
  4. Publish: A publish-binaries job copies the binaries into the plugforge repository under gm-build-latest/gm-cc/bin/ and updates the plugkitVersion field in gm.json. The commit does not carry [skip ci], triggering the next step.
  5. Plugin: plugforge’s publish workflow builds all 11 platform plugins and pushes to the downstream gm-cc repository.
  6. Delivery: The /plugin command in Claude Code tracks the HEAD hash of gm-cc; when it changes, the plugin cache is invalidated and the bootstrap script downloads the new binary on next agent start.
Observation (End-to-End Latency). From a push to rs-exec to binary availability in a running agent, the pipeline takes approximately 12–18 minutes, dominated by the six-platform Rust build. No human action is required at any step.

4.3 Force-Add Workaround

The gm-build-latest/ directory is listed in plugforge’s .gitignore to prevent accidental committing of local build artifacts. The publish-binaries job must use git add -f to force-add the binaries past this ignore rule. Without -f, the git add silently succeeds (exit 0) while staging nothing, and the subsequent commit contains no binary changes.

4.4 E403 Race Condition Tolerance

The cascade sometimes triggers two parallel npm publish attempts for the same version: once from plugforge’s publish workflow, and again from the downstream gm-cc repository’s own publish-npm workflow triggered by the push. The second publish always fails with E403 (cannot publish over a previously published version). Both workflows grep the npm error output for this specific message and treat it as a success condition rather than a failure. This makes the pipeline idempotent with respect to publish ordering.

5. Session Isolation and Process Hygiene

5.1 Compound Session Key

Background tasks are indexed by a compound key of session_id + cwd. This prevents two distinct agents operating in the same working directory from seeing each other’s task lists, while allowing a single agent to recover its own tasks after a network interruption. The compound key is forwarded by the pre-tool-use hook on every exec: invocation, ensuring the runner can always attribute an execution to its originating session.

5.2 Runner Start Lock

A file-system start lock prevents multiple concurrent plugkit runner instances. Previously, if the daemon failed to start and the agent retried, two runners would compete for the same port, producing split task state—each runner knowing about only half the active tasks. The lock is held for the duration of the runner process and released on exit or crash.

5.3 Orphan Process Reaper

Each task spawned by rs-exec runs as a child process in --exec-process-mode. When the runner restarts, its in-memory PID map is lost. Orphaned exec-process-mode children have no cleanup path: they hold file handles, ports, and browser sessions indefinitely. The reaper runs at startup and identifies orphans as processes with --exec-process-mode in their command line whose parent PID does not belong to a live runner-mode process, subject to a 5-second age guard that prevents the reaper from killing processes it just spawned.

In observed failure cases before the reaper was introduced, orphan accumulation reached 300+ processes consuming ~5.9 GB RAM. The reaper eliminates this failure mode by making startup the cleanup point rather than requiring operator intervention.

5.4 Process Tree Kill

Killing a task requires killing its entire process tree, not just the root PID. A child that spawns its own subprocesses (e.g., a Node.js script that forks workers) would otherwise leave zombie children after the parent is killed. The kill_pid function in rs-exec recursively enumerates the process tree using sysinfo and kills children depth-first before the parent, preventing orphan creation at kill time.

6. Observability Mandate

6.1 Permanent vs. Ad-Hoc Structures

The planning skill was extended with a mandatory observability enumeration pass executed on every planning iteration. The mandate distinguishes two categories of observability:

Permanent structures are named, queryable, and survive across agent interactions: a /debug/<subsystem> endpoint that returns current state, a window.__debug.<module> registry entry populated on mount and removed on unmount, a structured log with subsystem tag and severity filterable at runtime without restart.

Ad-hoc structures are not observability: a console.log statement, a temporary variable dump, a comment describing expected behavior. These require foreknowledge of where to look and disappear when the relevant code path is not executing.

6.2 Server-Side Requirements

Every internal subsystem must expose a named inspection endpoint. The mandate enumerates the required subjects: state machine phase, work queue depth, connection pool utilization, active task map, process registry, RPC handler latency, hook dispatcher invocation counts. State must be readable and filterable without restarting the daemon. Any internal state that requires a restart to inspect is classified as an observability gap and generates a .prd item in the next planning pass.

6.3 Client-Side Requirements

window.__debug must be a live structured registry. New modules register themselves on mount and deregister on unmount. The registry must be addressable by key: window.__debug.chat.pendingMessages, window.__debug.ws.connectionState, window.__debug.tasks.active. Any execution path not traceable through window.__debug is an observability gap.

6.4 Enforcement Mechanism

Observability improvements are classified as highest-priority items in the .prd and cannot be deferred. The planning skill is explicit: “on discovery of any observability gap, immediately add a .prd item.” This makes observability a first-class engineering concern that competes for priority with features rather than being deferred to a “logging pass” that never arrives.

7. Minimal-Code Structural Mandate

7.1 Resolution Order

Before writing any logic, the agent must follow a four-step resolution order, stopping at the first step that resolves the need:

  1. Native first: Does the language or runtime already do this? Use it as designed.
  2. Library second: Does an existing dependency solve this pattern? Use its API directly.
  3. Structure third: Can the problem be encoded as data (map, table, pipeline) so the structure enforces correctness and eliminates branching?
  4. Write last: Only author new logic when the above three are exhausted. New logic is new surface area and new bugs.

7.2 Structure as Correctness

The mandate elevates structure to a correctness argument, not a style preference. A dispatch table that maps tool names to handler functions cannot accidentally dispatch to the wrong handler—the table shape makes the wrong state structurally impossible. A pipeline that transforms data through a sequence of pure functions cannot accumulate mutable state in a loop variable—the pipeline shape eliminates the accumulator bug class. When a structural choice eliminates a class of wrong states, the mandate requires naming that property explicitly in code review, so future maintainers understand what the structure is protecting.

7.3 Readable Without Simulation

The mandate includes a readability criterion: code that can be read top-to-bottom without mental simulation is done right; code that requires the reader to trace execution flow to understand behavior is not. This operationalizes the principle without requiring subjective judgment about “clean code”—if you have to run the code in your head to know what it does, the structure needs to change.

8. Discussion and Limitations

8.1 Browser Runtime Scope

The managed browser runtime handles single-agent browser automation reliably. It does not address scenarios where multiple concurrent agents need to interact with different browser sessions on the same machine—the fixed port design assumes one active session per working directory. Multi-session parallelism would require dynamic port allocation with per-session port registration, which is straightforward to add but was not yet needed in our usage.

8.2 Vector Search Index Size

The flat vector index scales to medium codebases (tens of thousands of files) without issue but becomes slow for very large monorepos. An approximate nearest-neighbor index (e.g., HNSW) would be the natural next step, trading some recall for a logarithmic rather than linear query time. The nomic-embed-text-v1.5 model was chosen for its code retrieval performance; we have not systematically compared alternatives.

8.3 Cascade Pipeline Failure Modes

The cascade pipeline has two known failure modes. First, if the PUBLISHER_TOKEN secret is absent from any upstream repo, the cascade silently fails to trigger. Second, the cargo update step always pulls the latest git dependency hashes; if an upstream repo has a broken HEAD, the next cascade build fails until the upstream is fixed. Both are operational concerns rather than architectural ones, and both are visible in CI logs.

8.4 Observability Mandate Scope

The mandate requires permanent queryable structures but does not specify the query protocol. In our implementation, server-side inspection is HTTP-over-localhost; client-side inspection is browser console access to window.__debug. A more rigorous approach would define a structured inspection protocol (e.g., a JSON-RPC debug API) that tools can consume programmatically. The current approach is sufficient for interactive debugging but not for automated monitoring.

9. Related Work

Browser automation for testing and scraping is well-studied [2, 3], but its application to agent isolation—where the agent is the user and session leakage is the failure mode—is less documented. Playwright [2] provides the session isolation primitives we use; our contribution is the management layer that makes them automatic and transparent to the agent.

Hybrid dense-sparse retrieval [4] is established in information retrieval; our application combines BM25 with a quantized code-specific embedding model in a single search call, with a structured four-section output format designed for LLM consumption rather than ranked-list presentation. Git commit indexing as a search dimension is, to our knowledge, novel in this context.

Cascade deployment pipelines using GitHub Actions workflow_dispatch are standard CI/CD practice. Our contribution is the specific wiring that makes a single-repo code change propagate automatically through Rust binary builds, plugin packaging, and agent cache invalidation without manual version management.

The observability mandate draws on the structured logging and observability-as-code literature [5], but applies it specifically to LLM agent systems where the “operator” is the agent itself and the “system under observation” includes the agent’s own task state.

10. Conclusion

We have described a set of runtime infrastructure advances for the gm LLM coding agent orchestrator: a deterministic managed-browser runtime with per-session isolation and automatic process cleanup; a hybrid search backend combining BM25, vector embeddings, and git commit indexing; and a fully automated cascade deployment pipeline that propagates Rust library changes through binary builds to running agents without human intervention.

These advances address the operational layer below the behavioral envelope described in our earlier work. Lifecycle control rules are only as effective as the infrastructure that executes them. An agent with perfect planning discipline that cannot reliably launch a browser, search a codebase semantically, or run current code is still unreliable in practice. The infrastructure described here is the substrate on which behavioral correctness properties are expressed.

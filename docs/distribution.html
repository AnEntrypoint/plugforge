The SQLite Model:
Turnkey Binary, Open All The Way Down

AnEntrypoint
April 2026
Thesis
A coding agent should install in two commands and never need a build toolchain on the user’s machine. It should also have no proprietary core: every binary in the distribution must be built, in public, by CI, from source repositories anyone can fork. SQLite achieved this combination twenty-five years ago for an embedded database. gm applies the same combination to coding-agent infrastructure: the user gets a single trusted artifact that “just works,” and the artifact’s entire genealogy — every Rust crate, every workflow, every signing manifest — is open source under one organization, one cascade, one push at a time.

1. Why Distribution Is The Hard Part

Most coding agents fail at the install step. They ask for a Python virtualenv, a Node version, a Rust toolchain, an environment variable, a config file, an API key, and a directory layout the user has to memorize. By the time the agent runs, the user has spent more time on it than they intended to spend in a week. Worst, every component the agent needs is a separate vendor decision the user has to ratify implicitly.

SQLite’s design choice was: ship one C file. No toolchain, no daemon, no driver, no version skew. The user’s code links it; that’s the install. Two decades later, SQLite is in every phone, every browser, every aircraft. The lesson is not that databases should be embedded — the lesson is that the install experience is the product, and that radical simplicity at install is compatible with radical openness underneath.

For an agent, the equivalent is: a single binary, with a single hook into the host runtime, with all the heavy machinery (search, indexing, memory, runner, hook dispatch, browser automation) precompiled and signed in CI from open source repositories.

2. The Two-Command Install

Every supported platform installs the same way. For Claude Code:

claude plugin marketplace add AnEntrypoint/gm
claude plugin install -s user gm@gm

That’s the whole thing. No cargo, no npm install, no Python, no PATH edits. The first time the agent runs, the bootstrap script in bin/bootstrap.js resolves the platform-appropriate binary, fetches it from the public GitHub release, verifies its sha256 against a manifest checked into the plugin, and caches it under %LOCALAPPDATA%/plugkit/bin/v<version>/. Subsequent sessions skip the fetch and exec the cached binary directly.

The same shape works on the other eleven platforms (Gemini CLI, OpenCode, Kilo Code, Codex, Copilot CLI, Qwen Code, VS Code, Cursor, Zed, JetBrains, Hermes). Each has a thin platform-specific wrapper; the heavy artifact is the same Rust binary across all of them.

2.1 What “Turnkey” Excludes

  • No host toolchain. The user does not need cargo, rustup, cmake, gcc, or platform SDKs. They need only the host editor or CLI they were already going to use.
  • No native build step at install. Every artifact is precompiled by GitHub Actions runners against the four target tuples (win32-x64, linux-x64, linux-arm64, darwin-arm64; darwin-x64 and win32-arm64 available on demand) and uploaded to a public release.
  • No daemon to install or supervise. The runner spawns lazily on first exec:<lang> call, registers a port file in $TEMP, and is reused by all subsequent calls in the session.
  • No mandatory secrets at install. The agent runs offline against the local cached binary; the only network event at install time is the one-time fetch from the public release.

3. Open All The Way Down

The SQLite analogy holds on the openness axis only because every link in the chain is in public. There is no “and then a private build server takes over” step. The chain has six repositories under one GitHub organization (AnEntrypoint), and every one of them is MIT-licensed and forkable in its current state.

rs-execThe exec runner. Spawns subprocesses for nodejs, python, bash, typescript, go, rust, c, cpp, java, browser. Owns the background-task store, the port file, the kill-tree implementation, the timeout enforcement loop. The thing that turns a model’s “run this code” into a witnessed process exit.
rs-codeinsightThe code-search backend. Builds a per-project chunked index under .gm/code-search/, exposes exec:codesearch. Two-words-then-iterate query protocol; dense + symbol search blended; PDF pages indexed alongside source.
rs-searchThe file-system search backend. Path globs, content greps, type filters — all the operations that find + rg would cover, but called as one synchronous library function rather than two shelled-out processes.
rs-plugkitThe dispatcher. The single user-facing binary. Receives hook events (PreToolUse, PostToolUse, SessionStart, PromptSubmit, Stop, PreCompact), routes them to the appropriate handler, and embeds the runner, the indexer, and rs-learn. The Cargo.toml in this repo is the version source of truth for the entire distribution.
rs-learnThe memory layer. SQLite-backed graph of episodes, embeddings, and structured facts. Backs exec:recall on the read path and the memorize sub-agent on the write path. Per-project store under .gm/rs-learn.db; tracked in git so memory follows the project across machines.
gmThis repository. The orchestrator. Generates the fifteen downstream plugins from a single convention-driven source (gm-starter/). Owns the skill bundle, the prompts, the hooks, the bootstrap script, and the publish workflow that fans out to every platform.

4. The Cascade

Six repositories cooperate but no human steps between them. A push to any of the Rust-side repositories triggers a deterministic cascade that ends with a new plugin commit on every downstream:

rs-exec / rs-codeinsight / rs-search / rs-learn — push to main —> .github/workflows/cascade.yml fires rs-plugkit workflow_dispatch rs-plugkit — release.yml: cargo update, build 6 binaries, auto-bump patch version — publish-binaries job uploads to AnEntrypoint/plugkit-bin (sha256 manifest, plugkit.version) — bumps gm-starter/gm.json::plugkitVersion and gm-starter/bin/plugkit.version in AnEntrypoint/gm (NEVER pushes binaries into gm history) gm — that commit triggers gm publish.yml — 15-platform matrix: node cli.js gm-starter ./build (regenerates configs) copies plugkit.version + plugkit.sha256 into bin/ git init fresh empty dir + rsync + git add -A + git add -f bin/ diff --depth 1 against downstream (no-op detection) git push --force origin main (1-commit orphan history) downstream — gm-cc / gm-gc / gm-oc / gm-kilo / gm-codex / gm-qwen — gm-copilot-cli / gm-hermes — gm-vscode / gm-cursor / gm-zed / gm-jetbrains / gm-antigravity / gm-windsurf — each is one orphan commit, ~<1 MB — /plugin notices new HEAD → bootstrap.js fetches new binary

No human edits a version string. No human runs cargo build. No human syncs a downstream branch. The cascade is the publishing pipeline. The phrase “to update anything, push to the relevant repo” is literal: a one-line fix in rs-codeinsight propagates to fifteen plugin repos within one CI cycle.

5. Why The Downstreams Are Orphan-Committed

Each of the fifteen downstream repos is reset to a single orphan commit on every publish. This is unusual; the more common pattern is to append commits and let downstream repos grow. We chose orphan commits because:

  • Each downstream is a build artifact, not a project. Its history is the orchestrator’s history, not its own. Letting it accumulate per-publish commits would conflate build output with development.
  • The download is the product. A user who clones gm-cc wants the latest plugin, not its retroactive build log. The single-commit form makes the clone cheap (under 1 MB) and the contents trivial to diff against the next publish.
  • Force-push is the publishing primitive. Each publish is a fresh empty git init, an rsync of the build output, and a git push --force. There is no merge, no rebase, no history to preserve. The downstream is a one-frame snapshot that the orchestrator overwrites on every cascade tick.

5.1 The Binary Is Never In gm History

A subtle but important rule: the multi-megabyte plugkit binary lives only in the AnEntrypoint/plugkit-bin Releases. It is fetched by bootstrap.js at first use and cached locally. It is never committed to gm, and never to any downstream plugin repo. The plugin repo carries a bin/plugkit.version + bin/plugkit.sha256 manifest and the bootstrap script. This keeps every plugin repo small, makes the orchestrator’s git history fast to clone, and decouples plugin churn from binary churn.

6. The Bootstrap Cache

The first time any platform invokes the plugin’s plugkit entry point, bin/bootstrap.js resolves the binary. Its responsibilities, in order:

  1. Read bin/plugkit.version for the required version.
  2. Resolve the per-platform cache directory: %LOCALAPPDATA%/plugkit/bin/ on Windows, ~/Library/Caches/plugkit/bin/ on macOS, $XDG_CACHE_HOME/plugkit/bin/ elsewhere. Fall back to $TMPDIR/plugkit-cache/bin/ if the primary cache is unwritable.
  3. If v<version>/<binary> exists alongside an .ok sentinel, use it. No network call.
  4. Otherwise, acquire a per-version lock, fetch from github.com/AnEntrypoint/plugkit-bin/releases/download/v<version>/<binary>, verify the sha256 against the embedded manifest, atomically rename into place, write the .ok sentinel, prune older versions.

The fetch path is hardened against the failure modes encountered during early production: a 60-second per-attempt socket timeout, a 20-second stall watchdog that fires when the response stream stops producing bytes (a TCP connection can stay open and idle indefinitely otherwise), five attempts with backoff, a 5-minute lock TTL so a stale lockfile from a crashed install does not block forever, and HTTP Range-resumable downloads so a partially complete fetch survives a retry. These details are not user-visible; the user sees a small progress line on first run and never sees it again.

7. The 15-Platform Matrix

The same core (Rust binary, skills, prompts, hooks, lang plugins) ships to fifteen hosts. Each host has a different conception of what a “plugin” is — a Claude Code plugin is a marketplace package, a VS Code extension is a VSIX, a JetBrains plugin is a ZIP, OpenCode wants a directory, Codex wants a config block. The closest cousin is the SQLite amalgamation: one artifact, no install scaffolding, transparent provenance. The differences are that gm’s artifact is a precompiled native binary rather than a source amalgamation (because the user is not compiling against it; the user is exec-ing it), and that updates arrive via a versioned cache directory rather than a swap of one file.

9. Trust And Verifiability

A turnkey binary distribution buys convenience by trusting whoever built the binary. We make this trust auditable, not implicit:

  • Public build pipeline. Every binary is built by a GitHub Actions workflow whose YAML is in the same repository as the source. There is no offline build step. plugkit-bin Releases are produced by rs-plugkit CI; the run logs are public.
  • SHA-256 manifest. The plugin ships bin/plugkit.sha256 committed alongside bin/plugkit.version. The bootstrap verifies the downloaded binary against this manifest before caching. A binary whose hash does not match is rejected and the partial file is removed.
  • Reproducible enough. Linux and macOS builds are reproducible per-toolchain; Windows MSVC builds are not bit-reproducible due to PE timestamps but their hashes are still anchored at the plugkit.sha256 commit, so a man-in-the-middle attempt would fail verification before exec.
  • Forkable. Any user who does not trust the published binaries can fork all six repositories, point RELEASE_REPO in bootstrap.js at their own fork, and ship a private cascade with the same shape. Nothing in the distribution requires AnEntrypoint specifically.

10. Why Not Just Publish To npm / cargo / pip?

A reasonable question. Three reasons:

  1. Multiple language ecosystems. The agent has to live inside Claude Code (a Node.js host), Gemini CLI (Node.js), OpenCode (Node.js), VS Code (Node.js / VS Code API), and JetBrains (JVM / IntelliJ API), among others. There is no single registry that delivers to all of them. Every platform has its own plugin shape; the cascade builds each shape from one source.
  2. Toolchain absence at install. Many users install Claude Code with no Node toolchain, no Rust toolchain, no Python. npm install at agent install time would fail; cargo install would require rustup. The bootstrap-fetch model assumes only that the host can fetch() over HTTPS, which is universally true.
  3. Binary size. The plugkit binary is ~104 MB. Pushing that into npm, much less fifteen plugin repos, would inflate clone times and registry storage out of proportion to the install benefit. The Releases-bucket model keeps binaries out of git history while keeping them addressable by version.

11. Operational Lessons

Three production failures shaped the current pipeline:

  • Stalled fetches looked like success. An early bootstrap had only an HTTP socket-idle timeout. A response stream that opens, gets a few bytes, and then stalls held the connection for thirty minutes before the timeout fired. The stall watchdog — a 20-second timer reset on every chunk — closes this gap.
  • Hooks without timeouts spawned process avalanches. Pre-timeout-enforcement, a hook that called the runner without a budget could leak children indefinitely under load. The execute RPC now applies a 300-second default with a stderr deprecation warning when called without an explicit budget, and the CLI exec / bash subcommands reject missing or zero --timeout outright.
  • Concurrent CI publishes raced on the same downstream branch. The publish workflow now uses a fresh-init + force-push pattern rather than fetch-then-merge, which is faster and immune to the “rejected: fetch first” class of CI race.

None of these failures were architectural — the architecture was correct. They were operational holes that the architecture made cheap to plug.

12. The Invariant

Distribution Invariant. A user installs gm with two commands and never sees a build step. Every artifact they execute was built by a public CI run from a public Rust source repository under a single GitHub organization, signed by SHA-256, fetched over HTTPS at first use, cached locally for subsequent use, and replaceable at any time by forking six repositories and redirecting one constant. There is no proprietary core, no private server, no locked-in vendor surface, and no toolchain on the user’s machine.

This is what we mean when we call gm “the SQLite of coding-agent infrastructure.” Not that it is a database; that it has SQLite’s ratio of install-experience-to-openness. Two commands in. Six repositories out. Nothing in between.

PROMPT-DRIVEN SKILL INVOCATION DESIGN SPECIFICATION
====================================================

EXECUTIVE SUMMARY
-----------------

This specification defines a new capability for plugforge skills: prompt-driven invocation 
with structured input/output mapping. Skills accept natural language prompts as primary 
input and produce structured, immediately-usable output. The interface is defined entirely 
in SKILL.md metadata headers—no skill invocation mechanism changes required.

Key principle: Skills become composable, chainable tools that agents can invoke without 
intermediate transformation. One skill's output feeds directly as another skill's input.


1. SKILL.md HEADER SPECIFICATION
=================================

Current Format (YAML frontmatter):
  ---
  name: code-search
  description: Semantic code search...
  allowed-tools: Bash(bun x codebasesearch*)
  ---

Extended Format (Proposed):
  ---
  name: code-search
  description: Semantic code search...
  category: exploration
  allowed-tools: Bash(bun x codebasesearch*)
  input-schema: { type: object, required: [prompt], properties: {...} }
  output-schema: { type: object, properties: {...} }
  output-format: json|text|markdown|table
  error-handling: { timeout: {...}, invalid-input: {...}, ... }
  ---

New Header Fields:

  Field               Type      Required  Purpose
  ---------------------------------------------------
  name                string    Yes       Unique skill identifier
  description         string    Yes       User-facing description
  category            string    No        Skill category (exploration|transformation|analysis|data)
  allowed-tools       string    Yes       Tools skill is permitted to invoke
  input-schema        object    No        JSON Schema defining input format
  output-schema       object    No        JSON Schema defining output format
  output-format       string    No        Output serialization (json|text|markdown|table)
  error-handling      object    No        Defines behavior for error conditions


2. INPUT SCHEMA SPECIFICATION
==============================

Universal Required Fields:

  All skills using prompt-driven invocation MUST include:
  {
    "type": "object",
    "required": ["prompt"],
    "properties": {
      "prompt": {
        "type": "string",
        "minLength": 1,
        "maxLength": 1000,
        "description": "Natural language input describing the task"
      }
    }
  }

Standard Optional Fields (MAY be included):

  "timeout": { type: integer, minimum: 1000, maximum: 60000, default: 30000 }
  "context": { type: object, description: "Environmental context" }
  "filter": { type: object, description: "Output filtering options" }
  "format": { type: string, enum: [json, yaml, text, markdown, table], default: json }

Validation Rules:

  1. prompt field is required — always present as primary input
  2. All properties must have descriptions — used by agents to understand valid inputs
  3. Defaults must be sensible — allow skill to work with minimal input
  4. Enum values must be exhaustive — no surprises for tool invocations
  5. minLength/maxLength must prevent abuse — protect against DoS via enormous inputs
  6. Type specificity — use integer not number for counts, enum not string for fixed choices
  7. Required array must list all mandatory fields
  8. No null defaults — always provide real values

Input Validation at Invocation Time:

  1. Parse user input/agent argument as JSON (if not already)
  2. Validate against input-schema
  3. If invalid → return error with schema violation details
  4. If valid → execute skill
  5. If required fields missing → return error listing missing fields


3. OUTPUT SCHEMA SPECIFICATION
==============================

Universal Response Envelope:

  All structured outputs follow this base format:
  {
    "status": "success|partial|empty|timeout|error",
    "results": [...],
    "meta": {
      "count": integer,
      "filtered": integer,
      "duration_ms": integer
    },
    "errors": [...]
  }

Status Values:
  - success: All results retrieved, no errors
  - partial: Some results retrieved, execution timed out or hit limits
  - empty: Query executed successfully but returned no matches
  - timeout: Execution exceeded max-time without results
  - error: Execution failed with fatal error

Required Fields in All Responses:
  - status: ALWAYS present
  - results: ALWAYS present (may be empty array)
  - meta: ALWAYS present (with at minimum count, filtered, duration_ms)

Optional Fields:
  - errors: ONLY present if errors occurred (may be alongside partial results)
  - context: Optionally present for code/documentation results
  - metadata: Optionally present for file/content metadata

Input Schema for code-search:

  {
    "type": "object",
    "required": ["prompt"],
    "properties": {
      "prompt": {
        "type": "string",
        "minLength": 3,

        "description": "Natural language search query"
      },
      "context": {
        "type": "object",
        "properties": {
          "path": { "type": "string", "description": "Directory to restrict search to" },

      "timeout": { "type": "integer", "minimum": 1000, "maximum": 30000, "default": 10000 }
    }
  }

Output Schema for code-search:

    "properties": {
      "status": { "enum": ["success", "partial", "empty", "timeout", "error"] },

              "type": "object",

        "type": "object",
        "required": ["query", "count", "duration_ms"],
        "properties": {

5. IMPLEMENTATION NOTES
=======================

No Skill Invocation Mechanism Changes Required:

  The existing Skill tool (Skill in Claude Code) already supports:
  - Invocation by name: Skill("code-search", args)
  - Arguments passed as plain text string

  NEW REQUIREMENT: Agents invoke skills with JSON-formatted arguments matching input-schema.

  Example invocation:
    await Skill("code-search", JSON.stringify({
      prompt: "where is the database connection created",
      filter: { max-results: 20 }
    }));

  The hook that parses skill arguments should:
  1. Detect if argument string is valid JSON
  2. Validate against skill's input-schema (from SKILL.md header)
  3. On validation failure: return error with schema violation details
  4. On validation success: pass parsed object to skill implementation

Skill Implementation Changes (Minimal):

  Existing skill implementations require NO CODE CHANGES:
  - code-search: Already invokes bun x codebasesearch with natural language prompt
  - planning: Constructs .prd JSON file (already structured output)
  - agent-browser: Executes browser commands (output is already structured)

  Skills only need to UPDATE SKILL.md headers to declare their I/O schema.

Backwards Compatibility (100%):

  - Existing bun x codebasesearch "query" invocations continue to work
  - Skill tool with plain text args continues to work
  - Skills without input-schema are treated as legacy, work as-is
  - New validation is additive only — only skills with defined input-schema are validated
  - Missing schema = skill works with existing behavior
  - ZERO breaking changes to existing skills or invocations


6. ERROR HANDLING STRATEGY
==========================

Input Validation Errors:

  When input fails schema validation, return:
  {
    "status": "error",
    "results": [],
    "meta": { "query": null, "count": 0, "filtered": 0, "duration_ms": 50 },
    "errors": [
      { "code": "SCHEMA_VIOLATION", "message": "Missing required field: 'prompt'" },
      { "code": "TYPE_MISMATCH", "message": "filter.max-results must be integer, got string" },
      { "code": "VALUE_OUT_OF_RANGE", "message": "filter.max-results=1000 exceeds maximum of 500" }
    ]
  }

Timeout Handling:

  if (elapsed >= timeout) {
    return {
      "status": "partial",
      "results": resultsCollectedSoFar,
      "meta": { "count", "filtered", "duration_ms" },
      "errors": [{ "code": "TIMEOUT", "message": "Exceeded timeout..." }]
    };
  }

Execution Errors:

  if (executionFailed) {
    return {
      "status": "error",
      "results": [],
      "meta": { "duration_ms" },
      "errors": [{
        "code": "EXECUTION_FAILED",
        "message": stderr || error.message
      }]
    };
  }

  - Execute existing bun x codebasesearch logic

8. ACCEPTANCE CRITERIA
======================

Schema Completeness:
  ✓ Input schema includes: prompt (required), timeout, context, filter (optional)
  ✓ Output schema includes: status, results, meta, errors (structured envelope)
  ✓ All fields have descriptions and type specifications
  ✓ Validation rules prevent abuse (max-results ≤ 500, timeout ≤ 30s, etc.)
  ✓ Error conditions documented (timeout, invalid-input, empty-results)

Backwards Compatibility:
  ✓ Existing bun x codebasesearch "query" invocations still work
  ✓ Skills without input-schema work as legacy tools
  ✓ No breaking changes to Skill tool interface
  ✓ Old-style plain text arguments still accepted

Agent Integration:
  ✓ Agents can read SKILL.md and understand schema
  ✓ Agents can construct valid invocations from schema
  ✓ Agents can parse structured output without transformation
  ✓ Chaining is possible: output of one skill → input of next

Implementation Status:
  ✓ code-search SKILL.md updated with full schema
  ✓ Output format defined as JSON with status/results/meta/errors
  ✓ No changes to skill invocation mechanism required
  ✓ Optional hook validation added (non-blocking)


9. FUTURE ENHANCEMENTS
======================

Automatic Input Validation Hooks:
  Hook could validate Skill invocations against declared schema before execution:
  
  async function validateSkillInvocation(skillName, args) {
    const skillFile = await findSkillFile(skillName);
    const schema = extractInputSchema(skillFile);
    const parsed = JSON.parse(args);
    const valid = ajv.validate(schema, parsed);
    if (!valid) return { error: ajv.errorsText() };
    return { valid: true, input: parsed };
  }

Schema Versioning:
  Allow skills to evolve their I/O:
  
  input-schema-version: "1.0"
  output-schema-version: "1.0"
  schema-changelog:
    "1.1": "Added filter.exclude-patterns field"

Skill Composition Templates:
  Define how skills can be chained:
  
  compatible-predecessors: [code-search]
  compatible-successors: [planning, code-gen]
  transforms-output: false

Performance Hints:
  Allow skills to declare performance characteristics:
  
  performance:
    typical-duration-ms: 500
    max-results: 10000
    supports-streaming: false

11. CONCLUSION
==============

This design enables prompt-driven skill invocation with zero breaking changes to existing
systems. Skills declare their contract via SKILL.md metadata. Agents read these schemas
and automatically construct valid invocations. Outputs are structured and composable.

Implementation requires only SKILL.md additions — no tool mechanism changes, no code
modifications to existing skills. The entire system remains backwards compatible while
enabling powerful new capability: skill chaining and structured agent workflows.
